## 1. Resolved decisions (ADRs)

| # | Decision | Rationale / notes |
|---|---|---|
| ADR-001 | **Rendering abstraction: wgpu.** Single GPU API across D3D12/Vulkan/Metal/GLES; WGSL becomes the shader language for free. | Alternatives (per-platform native, OpenGL) rejected: maintenance × shader-portability cost. |
| ADR-002 | **Two backends, CPU is the renderer of record.** tiny-skia CPU backend is deterministic and headless; goldens are exact on CPU, perceptual on GPU. | Determinism for agents beats raw speed in tests; GPU parity-tested against CPU. |
| ADR-003 | **Runtime dependency whitelist:** wgpu, winit, taffy, parley, swash, tiny-skia, lyon, kurbo, libloading, notify, serde + serde_json, smol_str, bitflags, tokio (rt for agent/dev-server only), tungstenite/tokio-tungstenite, jsonschema (dev), proptest/criterion/insta (dev). Pin minor versions at repo init; record exact versions here in §3 when locked. | Anything else = escalation. Keeps the build auditable and the binary small. |
| ADR-004 | **Layout: Taffy behind `lumen-layout` wrapper.** Extensions (baseline alignment, intrinsic sizing beyond Taffy) implemented in the wrapper, upstreamed opportunistically. | Wrapper keeps engine replaceable; no direct taffy types in public API. |
| ADR-005 | **Text: parley + swash; bundled Noto fonts for all tests; no system fonts in CI.** | Cross-platform shaping determinism; system-font rendering is a headed-mode concern. |
| ADR-006 | **Path rendering v1: lyon tessellation → GPU triangles.** Vello-style compute rasterization is an M4 *evaluation*, not a dependency. | Lyon is boring and works on GLES-class hardware; revisit when scope allows. |
| ADR-007 | **Reactivity: fine-grained signals (Solid-style), no VDOM/diffing.** Identity = call-site path + explicit keys; state in a central serializable store. | O(changed) updates; the store is what makes hot reload & traces possible. |
| ADR-008 | **Tree + SoA hybrid, not ECS.** Widget logic in a tree; per-frame hot data in parallel arrays by NodeIndex; hit-test/cull are array scans. | UI workloads are sparse + hierarchical; this captures ECS cache wins where they matter without archetype churn. |
| ADR-009 | **Semantic tree = a11y tree = locator tree = agent tree.** One schema (03 §1), `semantics()` mandatory on leaf widgets. | Prevents drift between what tests query and what the AI sees. |
| ADR-010 | **Agent protocol: JSON-RPC 2.0 over WebSocket, mirrored as MCP tools; served by the CLI dev server, proxied to the app.** | One stable endpoint regardless of where the app runs (desktop/emulator/simulator). |
| ADR-011 | **State snapshots: serde_json (self-describing, field-tagged) in dev.** Positional/binary formats (postcard, bincode) forbidden for snapshots. | Survives struct evolution across hot reloads; size is irrelevant in dev. |
| ADR-012 | **Hot reload: three tiers** (data / cdylib build()-swap / snapshot restart); old dylibs never unloaded; abi_hash mismatch auto-downgrades tier 2→3. | Unloading is the segfault factory; leaking a few MB per swap in dev is fine. |
| ADR-013 | **No closures / fn-pointers / OS handles in stored state; trait objects only via `#[state_registry]`.** Handlers are re-created when their owning scope re-runs (reused from the F1 memo cache when a scope is skipped; retained on the node graph across pumps under F3) — never in stored state. *(Note amended 2026-07-03, §3; the hard precondition is unchanged.)* | Hard precondition for tiers 2–3, traces, and ADR-014. |
| ADR-014 | **Any-crate hot-patching linker is a separate future project, out of v1.** Lumen's only obligation: keep the `Checkpoint` protocol (02 §4) and ADR-013 discipline stable. | Subsecond/Live++-class function patching is proven feasible but is its own engineering effort. |
| ADR-015 | **Mobile: orchestrate official emulators** (avdmanager/adb, `xcrun simctl`), never ship our own. Physical iOS devices: tier-2 hot patch unsupported (codesigning) — tier 1 and 3 only. | |
| ADR-016 | **Styling: `.lss` files + 1:1 typed Rust mirror; parity enforced by `style_parity!` test.** CSS-like syntax chosen deliberately — maximal training-data familiarity for AI authors. | |
| ADR-017 | **Colors: f32 linear-light RGBA internally, sRGB/oklch at boundaries; gradients interpolate in Oklab.** | Modern interpolation quality; matches perceptual-diff metric. |
| ADR-018 | **Async: the framework core is single-threaded per window (no Send bounds on widgets); `resource()` futures run on a tokio runtime owned by the shell, results marshaled to the UI thread.** | Massively simplifies widget authoring; parallelism lives in render/layout internals where profitable. |
| ADR-019 | **Error codes are stable API** (E####/W####, registry in `lumen-core/diagnostics.md`). | Agents pattern-match on codes; never reuse or renumber. |
| ADR-020 | **License: MIT OR Apache-2.0; `cargo-deny` enforces compatible deps.** | |

## 2. Escalation list — do NOT decide these unilaterally
Stop the affected task, write `BLOCKED.md` (options + recommendation), continue elsewhere:
1. Any new runtime dependency outside ADR-003.
2. Any change to the schemas/grammars/protocols in docs 03–05 beyond additive optional fields.
3. Public API breaking changes after a milestone exit has been tagged.
4. Adopting Vello / replacing lyon (M4 evaluation outcome).
5. Physical-device (non-simulator) iOS/Android support scope.
6. Web/WASM target (explicitly out of scope for v1; tempting, refuse).
7. Multi-window APIs beyond a single primary window per app (design exists implicitly; defer).
8. Anything touching the security posture of the agent socket (auth model, non-loopback binds).

### T2.3 — Tier-2 hot patch (cdylib swap via libloading)
- Added `libloading 0.8` to the ADR-003 whitelist (workspace.dependencies) and to lumen-cli.
- `lumen-cli::hotpatch`: `HotComponent::load`/`swap` load a component cdylib, read its `lumen_abi_hash`/`lumen_build_label` C-ABI symbols, and hot-swap in place. Superseded libraries are *retired* (kept alive, never dlclose'd) to avoid use-after-free on live code/rodata pointers.
- State preservation falls out of the architecture: application state lives in the host-owned `lumen_core::Runtime`, never in the dylib, so reloading the component cannot touch it.
- ABI compatibility: a candidate whose `lumen_abi_hash` != `HOST_ABI_HASH` returns `Swap::NeedsTier3 { host, found }` — the caller downgrades to a tier-3 snapshot restart (T2.4) instead of swapping.
- Fixtures: crates/fixtures/{hot_a,hot_b,hot_c} are minimal cdylibs — hot_a/hot_b model a build() edit (Count -> Counter, same ABI), hot_c carries a different ABI hash.
- Tests (cargo test -p lumen-cli --test hotpatch): the libloading swap is sub-millisecond (well under the 2 s budget), host `count` signal survives the swap, the old lib is retired; the ABI-mismatch case rejects the swap and asks for tier 3.

### T2.4 — Tier-3 snapshot restart
- `AppSnapshot` (lumen-widgets::app) bundles the reactive store (`Runtime::snapshot`, ADR-011 self-describing field-tagged JSON) plus `focused` (host state, not in the store). It is `Serialize`/`Deserialize`, so it can be written before a kill and restored after.
- Scroll offset needs no special handling: the scroll widget keeps its offset in a `cx.signal`, so it is already part of the reactive store and rides along in the snapshot.
- `App::run_headless_restored(size, snap)` stages the snapshot via `Runtime::load_pending` *before* the first build so each signal adopts its restored value as it is recreated (Checkpoint protocol), then returns the instance plus `finish_restore` (W0002) drop diagnostics. `run_headless` now delegates to a shared private `boot`.
- Added `serde` to lumen-widgets (for the derive); exported `AppSnapshot` from lumen-widgets + the `lumen` facade.
- Tests (cargo test -p lumen-widgets --test tier3): drive count=5 + focus + scroll=40, snapshot → serialize → drop → deserialize → `run_headless_restored`; the re-serialized snapshot equals the original (signals+scroll+focus preserved), restore raises no drop diagnostics, and the restored state drives the live semantics. A guard test confirms a no-snapshot boot is at defaults.

### T3.5 — Gestures + mobile widgets + touch-target audit

### E8.9–E8.11 — time-driven animation (deterministic virtual clock)
- The runtime models time as **data** (`Headless::clock_ms`/`now_ms`), so animation is just "an input that arrives on a schedule." ### Extensibility for advanced widgets (charts) — Frame text + constraint-aware leaf measure
- **Motivation:** stress-test how far the flat `Element` + `NodeContent::Custom(Rc<dyn LeafWidget>)` (E2) seam carries for data-viz widgets. Two gaps blocked real charts; both are now fixed as a vertical slice (`examples/chart` proves it).
- **`Frame::fill_text` (lumen-render `canvas.rs`).** The draw `Frame` could only emit geometry, so a canvas/leaf couldn't label axes/bars. `lumen-render` can't depend on `lumen-text` (that would be a cycle — `lumen-text` depends on `lumen-render`), so `Frame` can't shape glyphs itself. Instead `fill_text(pos, text, TextOpts)` records a **deferred `FrameText` intent** (anchor point with the frame transform applied, plus `TextOpts{size,weight,color,AnchorX,AnchorY}`); `Frame::into_parts()` returns `(cmds, texts)`. The widget runtime resolves intents *after* the draw closure via `app::rasterize_frame_text`, which lays out + rasterizes through the existing `TextEngine` and shares the own-text glyph cache. It's a free fn over just the two fields it needs (`&mut self.text`, `&mut self.text_cache`) so it coexists with the `&self.meta` borrow the paint loop holds. Font size is **not** transform-scaled (translate-only placement); no glyph clipping yet.
- **Constraint-aware `Custom`-leaf measure (`app.rs` build_node).** Previously a custom leaf was measured at `(∞,∞)` and the result *overwrote* its style size, so a leaf was always its intrinsic size and could never flex/fill. Now the measure receives the **explicit** `width`/`height` as available space (∞ only for `Auto`/`Percent` axes), and an axis is overwritten by the intrinsic result **only when it is `Auto`**. So `width: 100%` (or any explicit size) survives and the leaf fills its container; `Auto` leaves keep sizing to content (no goldens move). Same philosophy as the text-wrap fix: explicit size wins, otherwise size-to-content.
- **Proof (`examples/chart`).** A `LineChart` (filled area + line + dots + Y ticks + X labels) and `BarChart` (gradient bars + value/category labels) as `LeafWidget`s in a responsive dashboard (`width:100%` panels). `semantics()` makes each chart accessible + agent-/test-addressable. Smoke tests assert chart semantics, painted series colours, and that the leaf flexes to its panel width (locking the measure change). Note labels drawn via `fill_text` are *painted* (image blits), so they live in pixels, not the semantics tree — the leaf's `semantics()` label is the addressable description.
- **Still deferred (not needed for static charts):** surfacing `PushLayer` clip on `Frame` (clip series to plot area for zoom/pan), an in-leaf event/hit-test hook on `LeafWidget` (tooltips/brushing — today compose overlay `Element`s), multi-stop/radial gradients + stroke dash/caps on `Frame`, and a retained/cached per-leaf draw list for very large series.

### Glass material — `backdrop-filter` (blur + saturate)
- **Motivation:** can the framework do Apple-style frosted glass? It had translucency, layer compositing, blend modes, gradients, and rounded-rect clip — but no **backdrop blur**, the defining piece. Added as a vertical slice (`examples/glass` proves it); refraction ("Liquid Glass" lensing) is backlogged.
- **Blur primitive (lumen-render `image.rs`).** `RgbaImage::blurred(radius)` — three box-blur passes (separable running-sum, edges clamped) ≈ Gaussian; `RgbaImage::saturate(factor)` scales colour around per-pixel luma. Reusable: the **drop shadow** now rasterizes its shape once and `blurred()`s it into a soft penumbra instead of stacking 8 translucent rects (no golden moved — no test exercises shadows; example pixel-tests are tolerant).
- **Backdrop render op (`DrawCmd::BackdropFilter { rect, radii, blur, saturate }`).** Implemented on the CPU renderer (the contract): map the region to physical px, snapshot the current target, crop a padded region, `blurred()` + `saturate()`, and composite it back clipped to the rounded rect — *before* the node's translucent fill. So a node with a translucent background over a `backdrop-filter` reads as frosted glass. The GPU backend leaves it to its existing `_ =>` no-op (it only does Rect/Image today), consistent with gradients/layers being GPU-deferred.
- **Cascade wiring (point 3).** `Style` gains `backdrop_blur` / `backdrop_saturate`; the `.lss` parser reads `backdrop-filter: blur(<px>) saturate(<n>|<pct>)` (the property name was already whitelisted but inert). The paint loop emits `BackdropFilter` for any node whose cascade sets a blur, right after the shadow and before the fill. So glass is authored purely in `.lss` (`background: #ffffff26; backdrop-filter: blur(18px) saturate(140%)`).
- **Proof + tests (`examples/glass`).** A vivid painted backdrop (gradient + saturated blobs) behind a translucent card that blurs it, plus pill "chips" with a lighter blur (the effect composes per node). Render tests in `lumen-render/tests/backdrop.rs` are the rigorous proof: `blurred()` turns a hard black/white seam into a ramp, and `BackdropFilter` softens a painted seam **only inside its region**. Smoke test asserts the card centre is a desaturated, lightened blend.
- **Backlogged (point 4):** refraction/lensing displacement + moving specular highlight (the full Liquid-Glass look), GPU-backend `BackdropFilter`, and a hairline rim border (borders aren't plumbed from `.lss`/`Element` yet). See `docs/backlog.md`.

### Renderer generics + the async/data layer (executor)
- **Renderer → defaulted generic.** Replaced `Headless { renderer: Box<dyn Renderer> }` with `App<R = CpuRenderer>` / `Headless<R = CpuRenderer>` (renderer stored as a value; `App::with_renderer` typestate builder). A blanket `impl<R: Renderer + ?Sized> Renderer for Box<R>` keeps dynamic dispatch as a *consumer opt-in* (`R = Box<dyn Renderer>`), so the previous boxed usage is a non-breaking reframing, not a removal. Defaulted params keep all existing `Headless` users compiling; `lumen-agent` is generic pass-through over `<R, E>`; the shell pins concrete types. CPU goldens untouched (default path unchanged). Rationale (with the user): the renderer's concrete type is consumed *runtime-internally* (in paint), so the generic flows usefully; runtime hot-swap is dropped (construction-time selection; the boxed type stays swappable since its type is stable).
- **Why the executor is *also* a clean generic (not erased).** Initial worry: an executor generic would infect `BuildCx` (hence user `build`/`#[component]` signatures). Resolved by deferring dispatch like `FrameRequests`: `cx.resource`/`cx.task` only *record* a `TaskRequest` during build; `pump()` dispatches it *after* the build (the runtime owns the executor + the deferred-op channel). The `Sink` handed to tasks carries only a channel `Sender` + optional waker — no executor type — so user fetchers never name `E`. Thus the executor is consumed only runtime-internally too ⇒ `Headless<R = CpuRenderer, E = InlineSpawner>` with the same defaulted-generic + blanket-`Box` pattern as the renderer.
- **Determinism preserved.** `Runtime` is `Rc<RefCell<…>>` (**not** `Send`), so background work never touches the store: it pushes a `DeferredOp` (`Box<dyn FnOnce(&Runtime) + Send>`) onto a channel; `Runtime::drain_deferred` (top of `pump`) applies them on the UI thread. `pump` stays a pure function of (state, queued events + deferred ops, clock). `Signal<T>` is a `Send` `u32` handle, so a task captures a signal + a `T: Send` value and the op does `sig.set(rt, v)` on the UI thread. Value-based `Sink::set`/`push` are recordable/replayable; `Sink::mutate(closure)` is the documented non-replayable hatch.
- **API.** `lumen_core::tasks`: `Spawner` (object-safe) + `InlineSpawner` (default, deterministic — inline blocking + hand-rolled `block_on`), `ManualSpawner` (records jobs; `run_pending` to step loading states in tests), `ThreadPoolSpawner` (std threads, no deps, `cfg(not(wasm32))`). `lumen_widgets`: `Resource<T,E>` (data+flags — `value`/`error`/`loading`, stale-while-revalidate), `cx.resource`/`resource_blocking` (keyed, dep-hash-tracked, generation-guarded cancel), `cx.task`/`task_blocking` (streaming). Shell upgrades the live app to `ThreadPoolSpawner` and wires a `ShellEvent::Wake` waker (via `EventLoopProxy` + `Headless::set_waker`) so a background result pumps + presents. `examples/data` demonstrates load + SWR refetch.
- **No new dependencies** for any of the above (pure std). **Deferred (Part D, the only ADR-003 escalation):** an async runtime + HTTP client (evaluate `ureq` blocking first) and a `WasmSpawner` (`wasm_bindgen_futures::spawn_local`) for browser parity. See `docs/plan-executor-and-renderer-generics.md` + `docs/backlog.md`.

### Rendering & performance plan (docs/plan-rendering-performance.md)
- 2026-06-24 — R0 — **Rendering regression harness.** `crates/lumen-render/tests/common` adds a reusable corpus (one display list per `DrawCmd` class), a perceptual diff (`assert_frames_close`, Oklab ΔE) and a byte-exact diff (`assert_frames_exact`), plus a **capability ratchet** (`gpu_supported(Cap)`): `cpu_vs_gpu` asserts GPU↔CPU parity for exactly the command classes the GPU claims, and each R1 sub-phase flips one `Cap`. `damage_equivalence` enforces `render_damage == full render cropped`. Calibration note: `Color::delta_e_oklab` is *unscaled* Euclidean Oklab distance (~0–1.5, JND≈0.02), so the old `gpu_parity` threshold of ΔE 2.0 was a no-op — replaced by `Tolerance::PARITY` (ΔE 0.04 / 0.5% of px) for edge-free content and `Tolerance::AA` (ΔE 0.04 / 4% of px) for anti-aliased content; removed the superseded `gpu_parity.rs`. The GPU differential self-skips (logging) when no wgpu adapter is present.
- 2026-06-24 — R1.2 — **GPU rounded rects + centered borders.** All solid-fill rects render through a per-corner rounded-box SDF fragment shader with 1px analytic AA; borders are a centered band straddling the path edge (matching tiny-skia's centered stroke). Axis-aligned square solid rects + nearest images stay byte-identical to CPU; rounded rects match within the AA budget (~0.9% of px on the seam).
- 2026-06-24 — R1.3 — **GPU paths via `lyon` + MSAA.** Locked **`lyon 1.0`** (ADR-003 whitelist / ADR-006). Solid-fill/stroke `DrawCmd::Path` is tessellated by lyon (`FillTessellator` NonZero / `StrokeTessellator`, tolerance 0.05) into a shared triangle buffer; AA comes from an MSAA offscreen target (best of 4×/2×/1× the adapter supports, resolved to the readback texture). The SDF rect fill is alpha-coverage based so MSAA leaves rects/images byte-identical. Paths match CPU within the AA budget (~2.4% of px on seams). *Known limitation:* draws are still batched by type (rects → paths → images) rather than strictly following display-list order; true interleaving lands with the layer work (R1.5).
- 2026-06-24 — R1.4 — **GPU gradients via Oklab ramp texture.** A shared `gradient` module exposes `sample_stops_oklab` (moved out of `cpu.rs`, which now reuses it for conic fills) + `bake_ramp`. The GPU bakes each gradient's stops into a 512-texel 1-D ramp texture using the *same* Oklab sampler, so colors match the CPU exactly; only the spatial parameter `t` (linear: projection; radial: distance/radius; conic: angle) is computed in the fragment, with pad/repeat/reflect spread. Solid-fill rects still go through the SDF pipeline; gradient-fill rects through the new gradient pipeline (square corners for now — rounded gradient rects later). Parity: max ΔE ≤ 0.005 across linear/radial/conic (well under the 0.04 ceiling).
- 2026-06-24 — R1.5 — **GPU layers (clip/opacity) + gamma-space blending.** `PushLayer`/`PopLayer` now render via a recursive `encode_layer`: each layer renders into its own MSAA→resolved texture (children recorded first), then composites over the parent with a rounded-rect SDF clip + group opacity (premultiplied source-over). Crucially, to match the tiny-skia CPU reference — which blends premultiplied **sRGB** directly — the GPU target changed from `Rgba8UnormSrgb` to **`Rgba8Unorm` holding sRGB-encoded bytes**: fragments sRGB-encode their linear colors (`lin_to_srgb`), and image/ramp textures store raw sRGB sampled without decode, so all fixed-function blending happens in gamma space. This dropped translucent-layer divergence from ~35% of pixels (ΔE 0.12) to 0.38% (ΔE 0.022, just the clip AA seam) while keeping opaque rects/images byte-identical. *Known limitations:* only source-over blend (Multiply/Screen/Overlay/Darken/Lighten need programmable dst-read — deferred); non-identity layer transforms and rounded **gradient** rects deferred; intra-layer draws still batched by type (composites drawn last). `BackdropFilter` still CPU-only.
- 2026-06-24 — R1.6 — **GPU HiDPI scale.** `GpuRenderer::render_at_scale` threads a logical→physical `scale` into the viewport uniform; `to_ndc` multiplies positions by it and the composite shader samples the (physical-size) child texture at the scaled coordinate. SDF/MSAA AA stays in logical space (a ~2× wider seam at 2×, still within the AA budget). Geometry/stroke widths scale; gradient `t` and conic angle are scale-invariant (logical). A 2× parity test (`gpu_matches_cpu_at_2x` vs `cpu::render_scaled`) passes for every supported scene. **Also fixed a CPU reference bug:** `fill_conic` sampled device pixels against a *logical* center, so conic gradients were mis-centered at non-unit scale; now divides the sample point by `base`'s scale (identity at scale 1 → goldens/damage unchanged). R1 (GPU surface backend command coverage) is complete bar glyphs (R3) and shaders.
- 2026-06-24 — R4 (finding) — **Multi-threaded layout is constrained by taffy.** Layout is `taffy::TaffyTree::compute_layout` (ADR-004), a serial whole-tree solve that can't be parallelized across disjoint subtrees of one tree (shared cache, single `&mut`). R4's "the SoA design invites it" refers to `scene.rs` culling (already threaded), not layout. Real parallel layout needs independent regions split into separate `TaffyTree`s computed on scoped threads and stitched (a `lumen-widgets` layout-driver change), gated by a node-count threshold + a 1-vs-N byte-identical test — its own design task, not a quick win. Recorded in `docs/plan-rendering-performance.md` (R4).
- 2026-06-24 — R3.1 — **Per-glyph raster cache (lumen-text).** Text was rasterized whole-string into a sprite (cached per string in the widget layer); a 1-char change re-rendered every glyph. `render_inner` now caches the swash alpha bitmap per glyph in a thread-local keyed by (font index, data len, glyph id, size bits, embolden bits, variation-coords hash). The pen is snapped to whole px so a glyph's bitmap is position-independent → output is byte-identical (goldens unchanged). Cache bounded at 8192 glyphs. Tests assert a changed string only rasterizes new glyphs and the cached path is byte-identical. Foundation for R3.2–R3.5 (GPU atlas + instanced `GlyphRun`).
- 2026-06-24 — R2 — **Damage-driven incremental repaint (paint side).** The runtime retains the previous frame's `DisplayList`; `paint()` computes `lumen_render::damage_between(prev, next)` → `Damage::{None, Region, Full}` (trim common content-equal prefix/suffix — image cmds compared by referenced bytes, not just index — then union the changed commands' `paint_bounds`). `None` reuses the frame (idle costs no raster); `Region` re-renders only that rect via the new `Renderer::render_damage` (CpuRenderer overrides with the deterministic `cpu::render_damage` at scale 1; default = full render + crop, always correct) and composites it into the retained frame via `RgbaImage::overwrite_rect`; `Full` repaints. The retained frame always equals a from-scratch render (R0 `damage_equivalence` + induction); `FrameStats.damage` + `force_full_repaint()` are exposed; the desktop shell skips present on an idle (`!painted`) tick. Regression tests in `lumen-widgets/tests/damage.rs`: idle paints nothing, a one-label edit damages only a bounded region (not Full), and the incremental composite is byte-identical to `force_full_repaint`. **Scope:** R2.1/R2.2 (dirty-flag marking + incremental `relayout_subtree`) are N/A under the full-rebuild reactive model (tree+layout recompute each pump) — damage is derived by DL diff instead; the dormant flags remain for a future incremental-tree model. Shell partial-tile texture upload deferred (Presenter retention refactor; minor since CPU raster is already saved).
- 2026-06-24 — R1.1 — **Live window renders through the GPU backend.** `lumen-shell` now uses the dynamic-renderer seam: `ShellRenderer = Box<dyn Renderer>`, chosen at startup (`GpuRenderer::new()` if an adapter is present, else `CpuRenderer`) via `App::with_renderer`. The GPU rasterizes the live window into the `Rgba8Unorm` gamma-space target (paths/gradients/layers/text-sprites all GPU-accelerated), reads back to the frame the always-on agent + the existing `Presenter` already need, and the presenter blits it (the present path was already a working wgpu surface). Verified by `lumen-widgets/tests/gpu_live_path.rs` (the exact boxed-GPU `App` construction matches CPU within the AA budget on a real screen, self-skips without an adapter) and a live launch (logs "GPU renderer active"; agent `ui.screenshot` returns a valid 440×520 frame). **Deferred (perf, not correctness):** zero-copy render-to-surface — the per-frame readback is currently required by the always-on live-window agent (it needs CPU pixels); a no-agent production build could skip it via device sharing + a surface blit pass. The format choice (`Rgba8Unorm` over a `Bgra8UnormSrgb` surface) is deliberate — it keeps gamma-space blending/CPU parity and matches the RGBA readback path; presenting to an sRGB surface stays correct through the presenter's existing linear-correct blit.
- 2026-06-24 — R1 (completion) — **Remaining GPU command coverage.** (1) *Draw order:* `encode_layer` now emits draws in display-list order (batching only consecutive solid rects / consecutive paths) instead of by type — fixes mixed-Z within a layer. (2) *Bilinear images:* `upload_image` selects nearest vs the linear sampler per `Filter` (was always nearest). (3) *Rounded gradient rects:* the gradient fragment clips to the rounded-box SDF (skipped when square, so square gradients stay byte-identical). (4) *Glass `BackdropFilter`:* implemented by splitting the layer pass at each backdrop op — render the segment, resolve, blur the resolved content (3 box passes per axis on Rgba8Unorm ping-pong textures, matching `RgbaImage::blurred`), then composite the blurred region back within the rounded clip with gamma-space saturation. Parity (cpu_vs_gpu): mixed_order/bilinear byte-identical at 1×; rounded gradient + backdrop within budget (backdrop max ΔE 0.027 at 1×). **Scoped out of R1 (no producer):** non-source-over blend modes (nothing emits a non-`SourceOver` `PushLayer`) and GPU `DrawCmd::Shader` (`ShaderWidget` pre-rasterizes to an `Image`, so the GPU never sees `DrawCmd::Shader`) — they remain CPU/display-list capabilities, to be added if/when a producer needs them. **Deferred (perf):** zero-copy render-to-surface (the live-window agent needs a CPU readback every frame; see R1.1).
- 2026-06-25 — R1 (revisit) — **GPU switched to linear-light blending (`Rgba8UnormSrgb`), reversing R1.5's gamma-space trick.** At the user's request: keep the `CpuRenderer` reference *unchanged* (gamma) but render the GPU in linear (the physically-correct blend, and what the live-window agent reads back). The target is now `Rgba8UnormSrgb`; solid fragments output linear color (hardware encodes), image/ramp/child textures are sRGB (sampling decodes to linear), readback returns sRGB bytes (RGBA, no swizzle). `lin_to_srgb` shader helper removed. **Consequence (accepted):** GPU and CPU now agree only on opaque, non-AA, nearest-sampled content; anti-aliased / blended / bilinear pixels diverge by design (linear vs gamma, e.g. layer ΔE ~0.12, bilinear ~0.06). `cpu_vs_gpu` reframed: `exact_vs_cpu(name)` (rect_solid, image_checker, mixed_order) keeps tight CPU parity; the rest are informational (logged ΔE) + checked for non-blank and GPU determinism (two renders byte-identical). The 2× test parity-checks only the nearest-image scene (SDF rects pick up sub-pixel AA at non-unit scale). Headless tests/goldens (CpuRenderer, gamma) therefore no longer pixel-match the live GPU on AA/blended content — the headless agent-gauntlet and the live agent differ there. Chose `Rgba8UnormSrgb` over `Bgra8UnormSrgb` (same linear result, no per-frame readback swizzle; BGRA only helps a future direct-to-surface path). *Rationale:* user wants the nicer linear blend as the live/agent truth without paying the CPU-renderer rewrite that full parity would require.
